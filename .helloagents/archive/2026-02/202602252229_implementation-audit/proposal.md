## 1. 需求

### 背景
本仓库实现了一个类似 Arthas 的 JVM 诊断工具（launcher + attach + 两段式 agent + 命令协议 + 插桩增强）。
随着功能面变大（watch/trace/monitor/tt/vmtool 等），“边界与生命周期”类缺陷（attach/detach 幂等、资源回收、协议与安全边界、配置漂移）
会成为最容易引发线上风险、也最难排查的隐患。

本次工作是一次**只读实现审计**：基于现有代码客观事实，输出“问题清单 + 证据定位”（P0/P1/P2 分级），不修改源码实现。

### 目标
- 梳理核心入口点与调用链（launcher→Attach→bootstrap agent→container/core→CommandProcessor→协议握手→命令执行）。
- 识别实现层面的潜在问题（bug、异常处理缺陷、并发/资源泄漏、性能隐患、安全与配置易错点、可维护性异味）。
- 形成可复核的问题清单：每条问题给出定位点（文件路径 + 类/方法）与简短证据说明。

### 约束条件
```yaml
时间约束: 单次审计在一个会话内完成
性能约束: 不运行压测；不对目标 JVM 做任何运行时操作
兼容性约束: 不引入新的构建/运行依赖
业务约束: 不修改任何源码实现（仅允许写入 .helloagents/ 方案包与归档）
```

### 验收标准
- [x] 输出 P0/P1/P2 问题清单（含证据定位点），覆盖 launcher/agent/container/core/foundation 关键链路。
- [x] 输出推荐优先深挖的热点文件建议（3–5 个）。

## 2. 方案

### 技术方案
采用“生命周期边界驱动”的审计路径（方案A，见下方技术决策）：
1) 以入口点与边界为主线建立调用链与不变量；2) 针对热点模块做定点深挖；3) 产出按严重级别分组的问题清单并给出证据锚点。

说明：本方案包类型为 overview，仅沉淀审计结论与建议，不包含开发实施任务。

### 影响范围
```yaml
涉及模块:
  - launcher: CLI 启动器、Attach、协议客户端
  - agent: bootstrap agent（premain/agentmain）、隔离加载与 gate
  - container: composition root（隔离 ClassLoader 内的 attach 生命周期）
  - core: transformer、命令服务端、命令执行管线、监控与 vmtool
  - foundation: 配置/Schema、安全与审计、通用工具
  - bootstrap: bootstrap 可见桥接与 SSOT 注册表
预计变更文件: 0（只读审计，无源码改动）
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 静态审计的误报/漏报 | 中 | 结论必须附带代码证据锚点；不确定项标注“需运行验证” |
| 结论依赖部署暴露面（本机 vs 端口转发） | 中 | 在问题条目中明确前置条件与影响范围（loopback/non-loopback、security.mode） |
| 子代理调用失败导致方案包填充降级 | 低 | 本次按“降级执行”由主代理直接填充并记录备注 |

## 5. 技术决策

> 本方案涉及的技术决策，归档后成为决策的唯一完整记录

### implementation-audit#D001: 采用“生命周期边界驱动”的只读实现审计作为输出主线
**日期**: 2026-02-25
**状态**: ✅采纳
**背景**: Java-Sleuth 采用“两段式 agent + 隔离 ClassLoader + 命令协议 + 插桩增强”，风险点集中在生命周期边界（attach/detach/re-attach）与跨边界 SSOT（bootstrap registry / sysprop rollback）。按边界审计更容易发现高影响缺陷并提供可复核证据。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 生命周期边界驱动审计（推荐） | 更贴合真实风险分布；证据链天然跨模块；结论更可行动 | 需要阅读更多关键链路代码 |
| B: 行为驱动闭环（证据卡片） | 更易落地成 QA/运维流程 | 静态审计阶段难以覆盖全部“行为证据”，容易偏流程而弱代码定位 |
**决策**: 选择方案 A
**理由**: 本次交付物是“只读问题清单 + 证据定位”，方案 A 能以更少噪声得到更高置信度结论。
**影响**: 不影响运行时行为；仅影响审计报告的组织方式与建议优先级排序。
