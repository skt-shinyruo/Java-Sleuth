# 技术设计：docs 文档中文化

## 技术方案

### 核心技术
- Markdown（GitHub/GitLab 常见渲染规则）

### 实现要点

1. **翻译范围**
   - 翻译：标题、段落、列表、表格说明文字、告警/提示文案
   - 保留：代码块（fenced code block）、命令行示例、配置 key、环境变量名、文件路径、类名/方法名等代码标识符

2. **链接与锚点稳定性**
   - 对带“目录（Table of Contents）”的文档，使用显式锚点（`<a id="..."></a>`）保持原有 `#overview` 等跳转入口不变
   - 目录文字翻译为中文，但锚点 id 维持稳定，避免外部引用失效

3. **一致性与可读性**
   - 术语尽量统一：如“部署”“运维”“排障”“监控”“可观测性”等
   - 保留产品名与命令名：Java-Sleuth、SleuthLauncher、`watch`、`trace` 等

## 安全与性能

- **安全：** 文档中不引入任何密钥/口令/真实联系方式；示例继续使用占位符或本地回环地址
- **性能：** 仅文档变更，无运行时性能影响

## 测试与交付

### 校验策略
- 使用 `rg` 粗查 `docs/` 下是否仍存在大段英文标题/说明（允许保留专有名词与代码块内容）
- 检查目录型文档的内部链接是否可正确定位（锚点存在且未重复）
